== Introduction

Meshed is a schedule optimization system that generates practice schedules for university athletic departments. It addresses scheduling needs of student-athletes, coaches, and facility managers. It also exhibits an opportunity for more generalized scheduling use-cases such as shift-work, recreation leagues, and more.

The purpose of this document is to provide a structured architecture description of Meshed following the arc42 template. By utilizing the C4 model, this documentation aims to create a clear and standardized representation of the system architecture for developers, architects, and stakeholders.

### Goals of the Documentation
The primary goals of these documents are to:

- Provide a structured and detailed overview of Meshed's architecture.
- Ensure a clear understanding of system interactions, components, and deployment strategies.
- Facilitate maintainability and scalability by documenting key architectural decisions.
- Enhance security and compliance by outlining specific choices made by the team.

### 1.1 Requirements Overview
The most important user requirements Meshed implements: 
- The system shall generate optimized practice schedules efficiently based on athlete availability.  
- The system shall clearly present facility booking conflicts in multiple formats. 
- The system shall provide an intuitive interface for viewing, editing, and managing personal and team schedules.

A full list of user requirements is provided in the docs/SuplementaryMaterials folder.

### 1.2 Quality Goals
The most important engineering specifications Meshed implements include:

| ID | Description |
|----|-------------|
| 1  | The scheduling algorithm must generate a weekly team practice for up to 150 users within 2 seconds in 95% of cases. |
| 2  | The database must return all events for a single team within a 7-day window in ≤ 600 ms with up to 5000 stored events. |
| 3  | The system must parse an uploaded .ics file with user events with a 95th-percentile latency of no more than 3 seconds. |

A full list of user requiremtns is provided in the docs/SuplementaryMaterials folder.
