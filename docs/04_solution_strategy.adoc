
## Solution Strategy

Meshed is a MVC, cloud deployed, web app. The system is encapsulated in a gradle environment to ensure quick and intuitive launching.

This document outlines the major decisions made in Meshed's architecture, technology stack, and CI/CD practices

### 4.1 Architectural Approach

**Model View Controller Architecture**: Follows an MVC approach **Model, View, and Controller aspects**.
Rationale: Simplifies adding new functionalities due to its modular nature. Easy to learn. Other possibilities included microservices. These two architectures have well developed migration plans. Microservices, being more complicated, was chosen as a possible future implementation.

### 4.2 Technology Stack

**Backend**: TypeScript.
- Justification: Ease of implementation with React, flexibility.
**Frontend**: React for web application, TypeScript.
- Justification: Industry standard React, ease of implementation with TypeScript
**Database**: PostgreSQL.
- Justification: Relational DB requirements, familiarity.
**Security**: BCrypt for password hashing.
- Justification: ISO and other requlation compliancy.
**Cloud**: Google suite (Firebase, and Google Cloud) for infrastructure.
- Justification: Low price, student credits, accessibility.
**Infrastructure**: Dockerized containers for local hosting of the prototype.
- Justification: Ease of development, launching, implementing changes.

### 4.3 CI/CD & DevOps Practices

- **CI/CD Pipelines**  Integrated to some extent in this application. Branching and pull request structure creates a simulated CI/CD process throughout development.
- **Linter** VS Code packages were installed to lint the code. It flagged programming errors and simple bugs.
- **Automated Testing**: The Google Cloud and CI/CD pipeline includes automatic tests that are run every time the backend is run locally.
